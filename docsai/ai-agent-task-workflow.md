# AI Agent Task Workflow (атомарные задачи + TDD)

## Зачем это нужно
Этот документ задаёт единый процесс постановки и выполнения задач AI‑агентом так, чтобы:
- изменения шли **маленькими атомарными порциями**,
- каждую порцию было **легко ревьюить**,
- каждую порцию можно было **покрыть тестами** (Pest) и вести через **TDD**,
- статанализ (PHPStan/Larastan, опционально Psalm) был “воротами качества”.

---

## Где живут документы
- Процесс (этот файл): `docsai/ai-agent-task-workflow.md`
- Спеки задач: `docsai/tasks/`
  - формат: `docsai/tasks/YYYY-MM-DD/<short-slug>.md`  
    пример: `docsai/tasks/2026-01-26/users-create-endpoint.md`

---

## Что такое “атомарная задача”
Атомарная задача — это изменение, которое:
- решает **одну понятную проблему**,
- трогает **минимально необходимое количество файлов**,
- легко откатывается,
- имеет чёткие **Acceptance Criteria** и **Definition of Done**,
- может быть проверено тестами и статанализом.

> Если задача “слишком большая”, она должна быть разбита на цепочку атомарных задач/PR.

---

## Контракт между человеком и AI‑агентом
AI‑агент обязан:
- следовать правилам проекта (типизация, тонкие контроллеры, FormRequest, именованные роуты, DDD-lite и т.д.)
- не делать оверинжиниринг (YAGNI)
- держать PR маленьким и логичным
- предлагать рефакторинг, если без него код нетестируем

Ревьюер (человек) обязан:
- ревьюить по спеке и чеклисту
- принимать/отклонять атомарно (без “ну давай ещё тут всё перепишем” в одном PR)

---

## Процесс выполнения задачи (строго по шагам)

### 1) Создать спецификацию задачи (Task Spec)
Перед кодом создаём файл в `docsai/tasks/YYYY-MM-DD/<short-slug>.md` по шаблону ниже.

В спеке обязательно:
- контекст
- цель
- границы
- критерии приёмки
- план реализации (минимальный)
- план тестов (желательно TDD)
- список затрагиваемых файлов/модулей
- риски и план отката

### 2) План: минимум кода, максимум ясности
- Начинаем с самого простого решения.
- Если нужно — добавляем DTO/VO/интерфейсы, но только по необходимости.
- Никаких “архитектур на будущее”.

### 3) Тесты (TDD по возможности)
- Сначала добавляем/обновляем тесты (Pest) на Domain/Application.
- Feature tests — только для “склейки” (endpoint → use-case), минимально.
- Не тестируем внешний мир напрямую.

### 4) Реализация
- Контроллер тонкий.
- Валидация в FormRequest.
- Бизнес‑логика в Application/Domain.
- DB/интеграции в Infrastructure.
- Роуты именованные всегда.

### 5) Статанализ и качество
Перед готовностью PR:
- PHPStan + Larastan: **0 новых ошибок**
- Psalm (если включён): **0 новых ошибок**
- Форматирование (Pint): ок
- Все тесты зелёные

### 6) PR и ревью
PR должен содержать:
- ссылку на спеки (`docsai/tasks/YYYY-MM-DD/<short-slug>.md`)
- краткое резюме
- как проверить (команды)
- какие тесты добавлены/изменены
- какие миграции/конфиги

---

## Чеклист ревью (быстро, но жёстко)

### Архитектура
- [ ] Контроллер тонкий
- [ ] Валидация только в FormRequest
- [ ] Логика в Application/Domain
- [ ] Infrastructure реализует интерфейсы
- [ ] Domain не зависит от Laravel

### Код‑стайл
- [ ] strict_types и типизация соблюдены
- [ ] нет лишней вложенности if/циклов
- [ ] нет if/elseif цепочек там, где нужен match
- [ ] нет массивов как коллекций в Application/Domain (границы допускают)
- [ ] исключения не используются как control flow (или обработка централизована)

### API
- [ ] единый контракт ошибок/статусов соблюдён
- [ ] JsonResource/mapper — только форматирование, без бизнес‑логики

### Роутинг
- [ ] каждый роут имеет name
- [ ] нет хардкодных URL строк в коде

### Качество
- [ ] тесты (Pest) покрывают домен/use‑case
- [ ] статанализ зелёный
- [ ] изменения атомарные и легко откатываются

## Шаблон спеки задачи (копировать в `docsai/tasks/YYYY-MM-DD/<short-slug>.md`)

```md
# <Название задачи>

## Контекст
- Что есть сейчас?
- Почему это проблема/что болит?

## Цель
- Что должно появиться/измениться?

## Не‑цели (Non-goals)
- Что точно НЕ делаем в рамках этой задачи (чтобы не расползалось)?

## Acceptance Criteria (критерии приёмки)
- [ ] ...
- [ ] ...
- [ ] ...

## Definition of Done
- [ ] Тесты (Pest) добавлены/обновлены
- [ ] PHPStan/Larastan без новых ошибок
- [ ] (Если включён) Psalm без новых ошибок
- [ ] Документация/контракты обновлены при необходимости
- [ ] Роуты именованы (если касается роутинга)

## Дизайн / решение (коротко)
- Куда кладём код (слои, классы)?
- Какие интерфейсы/DTO/VO нужны?

## План реализации (атомарно)
1. ...
2. ...
3. ...

## План тестов (TDD)
- Unit (Domain): ...
- Unit (Application/use-case): ...
- Feature (минимально): ...

## Затронутые файлы/модули
- `...`
- `...`

## Риски и откат
- Риски: ...
- Откат: ...
```

## Команды проверки (пример)
> адаптируйте под ваш composer.json

- Тесты: `composer test`
- Статанализ PHPStan: `composer stan`
- Psalm (если есть): `composer psalm`
- Форматирование: `composer pint`
