# Нормализация справочника категорий обращений до 15 категорий

## Context

В проекте уже есть публичный API `GET /api/v1/categories` и SSR-страница `/categories`.

Текущий backend read-model в `App\Application\PublicContent\PublicContentService::categoryGroups()` отдаёт 9 категорий в 3 группах. При этом frontend-мастер подачи обращения `frontend/app/pages/appeal/new.vue` содержит отдельный hardcoded-список из 15 категорий, часть slug-ов не совпадает с API:

- API использует `zhkh`, форма использует `housing`;
- API использует `citizen-rights`, форма использует `civil-rights`;
- API использует `veterans`, форма использует `svo-veterans`;
- часть категорий есть только во frontend-форме.

Из-за этого `/categories`, главная страница, фильтры обращений и мастер подачи обращения могут показывать разные справочники.

## Goal

Сделать единый справочник категорий обращений из 15 категорий и использовать его как источник истины для публичной страницы категорий и мастера подачи обращения.

Итоговый набор категорий:

1. ЖКХ
2. Дороги
3. Транспорт
4. Благоустройство
5. Здравоохранение
6. Образование
7. Социальная защита
8. Экология
9. Безопасность
10. Госуслуги и документы
11. Земля и недвижимость
12. Работа органов власти
13. Связь и интернет
14. Торговля и услуги
15. Другое

## Non-goals

- Не делать админку управления категориями.
- Не переносить категории в полноценные database CRUD-таблицы, если текущий seed/read-model достаточно для этапа проекта.
- Не менять механику отправки обращения за пределами выбора категории.
- Не добавлять многоязычность категорий.
- Не менять публичные URL страниц, кроме slug-ов категорий в query-параметрах.

## Frontend changes

- Убрать независимый hardcoded-список категорий из `frontend/app/pages/appeal/new.vue` или заменить его на данные из `GET /api/v1/categories`.
- Мастер подачи обращения должен принимать `?category=<slug>` только из API-справочника.
- Если query-параметр содержит неизвестный slug, выбирать первую категорию из API-ответа.
- Страница `/categories` должна показывать все 15 категорий.
- Блок категорий на главной странице должен использовать тот же набор slug-ов или явно показывать subset, совместимый с API.
- Типы `CategoryDto` / `CategoryGroupDto` должны остаться синхронными с API-контрактом.

## API contract

### `GET /api/v1/categories`

Route name: `api.categories.index`.

Success `200`:

```json
{
  "data": {
    "groups": [
      {
        "slug": "city",
        "title": "Дом и город",
        "categories": [
          {
            "id": "00000000-0000-4000-8000-000000000101",
            "slug": "zhkh",
            "title": "ЖКХ",
            "description": "Управляющие компании, подъезды, отопление, вода, лифты",
            "icon": "home"
          }
        ]
      }
    ],
    "seo": {
      "title": "Все категории обращений",
      "description": "Полный список категорий обращений и жалоб.",
      "canonicalUrl": "https://example.com/categories",
      "robots": "index,follow",
      "ogImageUrl": null,
      "lastModifiedAt": null
    }
  }
}
```

Контракт должен гарантировать:

- `groups` не пустой;
- суммарно в `groups[*].categories` ровно 15 категорий;
- `slug` уникален в рамках всего справочника;
- `id` уникален и стабилен;
- `seo` сохраняет текущую структуру.

## SEO impact

- `/categories` остаётся индексируемой SSR/SWR-страницей.
- Canonical остаётся без query-параметров.
- Изменение списка категорий не должно создавать отдельные индексируемые страницы категорий, пока такой маршрут не реализован отдельной задачей.
- `/appeal/new?category=<slug>` остаётся noindex, так как это форма подачи обращения.
