# Режимы подачи обращения: инкогнито и с контактами

## Context

Форма `/appeal/new` уже доступна гостям и авторизованным пользователям, умеет сохранять черновик, загружать вложения и отправлять обращение через существующие endpoints черновиков.

Новая бизнес-логика оставляет только два способа подачи:

- инкогнито;
- с контактными данными.

Любое обращение должно проходить проверку. Разница только в обязательных данных и последующем UX.

## Goal

Адаптировать frontend-форму подачи обращения под два режима:

- при подаче инкогнито требовать минимум одно фото, видео или документ;
- при подаче с контактными данными требовать ФИО и телефон или email;
- не предлагать личный кабинет при подаче инкогнито;
- сохранить общий pipeline отправки обращения без изменения API.

## Non-goals

- Не менять backend API.
- Не менять формат черновика обращения.
- Не редактировать `static/`.
- Не внедрять реальную SmartCaptcha.
- Не создавать автоматическую регистрацию пользователя.

## Backend changes

Backend не изменяется.

## Frontend changes

- Сценарий выбора способа подачи сокращён до двух вариантов.
- Удалён отдельный frontend-вариант "из личного кабинета".
- Авторизованный пользователь открывает форму в режиме "с контактными данными" с предзаполнением профиля.
- Для инкогнито-подачи контактные поля скрыты, контакты не отправляются в payload.
- Для инкогнито-подачи добавлена frontend-валидация обязательного вложения.
- Для подачи с контактными данными добавлена frontend-валидация ФИО и минимум одного канала связи.
- Предложение личного кабинета отображается только в контактном сценарии.
- Result screen показывает разные следующие шаги для инкогнито и контактной подачи.

## API contract

Контракт не меняется. Используются существующие endpoints:

- `POST /api/v1/appeal-drafts`
- `PATCH /api/v1/appeal-drafts/{id}`
- `POST /api/v1/appeal-drafts/{id}/attachments`
- `DELETE /api/v1/appeal-drafts/{id}/attachments/{attachment}`
- `POST /api/v1/appeal-drafts/{id}/submit`

При подаче инкогнито frontend отправляет контактные поля как `null` и `contact_visibility: hidden`.

## SEO impact

`/appeal/new` остаётся workflow-страницей с `noindex`. SEO-изменений нет.

## Tests

Проверить:

- отсутствие старых режимов `public` и `account` в форме;
- `pnpm typecheck` для Nuxt;
- `pnpm lint` для Nuxt, если окружение позволяет.

## Acceptance criteria

- [x] На форме доступны только два способа подачи: инкогнито и с контактными данными.
- [x] При инкогнито-подаче нельзя перейти дальше шага доказательств без вложения.
- [x] При инкогнито-подаче контакты не отображаются и не уходят в payload.
- [x] При инкогнито-подаче не показывается предложение создать личный кабинет.
- [x] При подаче с контактами обязательны ФИО и телефон или email.
- [x] Любой сценарий сообщает, что обращение будет проверено.

## Postman

Postman collection не требуется обновлять, потому что API endpoints и payload contract не менялись.
