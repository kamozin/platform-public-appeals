# Сохранение прочитанности уведомлений личного кабинета

## Context

В личном кабинете кнопка «Отметить прочитанными» вызывает API и меняет состояние на фронте, но после перезагрузки уведомления снова отображаются как новые. Backend возвращает системные уведомления с `read: false` и не сохраняет факт прочтения.

## Goal

Сохранять состояние прочитанности уведомлений отдельно для каждого пользователя.

## Non-goals

Не внедряем полноценный центр уведомлений, доставку push/email, удаление уведомлений или персональные тексты уведомлений.

## Backend changes

Добавить таблицу `user_notification_reads` и использовать её в `DashboardService` для расчёта поля `read`.

## Frontend changes

Новых изменений не требуется: текущий экран уже вызывает `POST /dashboard/notifications/mark-all-read` и обновляет локальный список.

## API contract

Существующие endpoints сохраняются:

- `GET /api/v1/dashboard/notifications`
- `POST /api/v1/dashboard/notifications/mark-all-read`

После вызова `mark-all-read` последующий `GET` возвращает уведомления с `read: true`.

## SEO impact

Нет. Личный кабинет закрыт от индексации.

## Tests

Обновить feature test личного кабинета: проверить, что уведомления сначала новые, после `mark-all-read` становятся прочитанными и остаются такими при повторном чтении.

## Acceptance criteria

- Уведомления отмечаются прочитанными после клика.
- Статус прочитанности сохраняется после повторного запроса.
- Состояние хранится отдельно для пользователя.

## Postman

Новые endpoints не добавляются. Коллекция Postman не меняется.
