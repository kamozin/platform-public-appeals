# Пользовательские Nuxt workflow

Nuxt получил страницы и формы для пользовательских сценариев. Часть страниц требует авторизацию, а форма `/appeal/new` доступна гостю и авторизованному пользователю.

## Страницы

- `/login` — вход.
- `/register` — регистрация.
- `/verification` — отправка и проверка кода.
- `/password-reset` — восстановление пароля.
- `/appeal/new` — мастер создания черновика обращения, доступен без авторизации.
- `/dashboard` — личный кабинет.

Все страницы используют `useNoindexSeo()` и исключены из `robots.txt`.

## Auth state

`useAuth()` хранит Bearer token в cookie `rd_auth_token` и пользователя в `useState`.

API-запросы идут через общий `useApi()`. Для приватных запросов добавляется header:

```txt
Authorization: Bearer <token>
```

Для гостевого черновика `/appeal/new` сохраняет token из `data.guestToken` в cookie `rd_appeal_draft_token` и передаёт его в дальнейших запросах:

```txt
X-Appeal-Draft-Token: <guest-token>
```

## Профиль и аватарка

В `/dashboard` профиль загружается через `GET /profile`, обновляется через `PATCH /profile`.

Аватарка обрабатывается на frontend до отправки:

- пользователь выбирает JPG, PNG или WebP до 10 МБ;
- browser canvas делает центральное квадратное кадрирование;
- изображение уменьшается до `512x512`;
- результат конвертируется в WebP и проверяется на лимит upload до 2 МБ;
- обработанный файл отправляется как `multipart/form-data` в `POST /profile/avatar`;
- удаление выполняется через `DELETE /profile/avatar`.

Frontend не хранит аватарку в `localStorage`: после успешного upload обновляются `profile.avatarUrl` и `auth.user`.

## Безопасность кабинета

Во вкладке `/dashboard` → `Безопасность` действия вынесены в отдельные карточки:

- смена пароля открывает модальное окно с текущим паролем, новым паролем и подтверждением;
- включение email-2FA открывает модальное окно с двухшаговым вводом: текущий пароль, затем код из письма;
- отключение email-2FA открывает отдельное подтверждающее модальное окно с текущим паролем.

Модальные окна используют общий `body.modal-open`, закрываются по `Escape` и возвращают результат в карточку действия.

## Комментарии и формы

Страница `/appeals/[slug]` получает комментарии через API и отправляет новый комментарий с pending state.

Форма `/appeal/new` поддерживает два способа подачи обращения:

- `Инкогнито` — контакты не запрашиваются и не отправляются, но обязательно минимум одно вложение: фото, видео или документ. Личный кабинет после такой подачи не предлагается.
- `С контактными данными` — обязательны ФИО и минимум один канал связи: телефон или email. Контакты используются для уточнений, проверки и выезда на место, но не отображаются в публичной части обращения.

Любое обращение отправляется на проверку независимо от выбранного способа подачи. Frontend не меняет API-контракт черновика: при инкогнито-подаче контактные поля уходят как `null`, `contact_visibility` остаётся `hidden`.

Блок "Место происшествия" использует клиентский компонент `AppealYandexLocationMap` на базе Yandex Maps JavaScript API 2.1. Для работы карты нужен публичный ключ:

```txt
NUXT_PUBLIC_YANDEX_MAPS_API_KEY=<ключ JavaScript API Яндекс.Карт>
```

Ключ передаётся в Nuxt через `runtimeConfig.public.yandexMapsApiKey`. Если ключ не задан или скрипт карты не загрузился, форма остаётся рабочей: пользователь может заполнить адрес вручную. В API по-прежнему отправляется только строка `location`; координаты в рамках текущего контракта не сохраняются.

Footer subscription и блок видео поддержки подключены к API:

- `POST /subscriptions/news`
- `POST /support-videos`

На главной странице отправка видео открывается через модальное окно в `HomeSupportVideo.vue`; форма остаётся SSR-safe и отправляет только `email` и `video`, как ожидает текущий API-контракт.

## SEO

`/robots.txt` создаётся Nitro route и закрывает:

- `/api`
- `/dashboard`
- `/login`
- `/register`
- `/password-reset`
- `/verification`
- `/appeal/new`

`/sitemap.xml` создаётся Nitro route и берёт URL из `GET /api/v1/seo/sitemap-urls`.
